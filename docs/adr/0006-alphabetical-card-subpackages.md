# Alphabetical sub-packages with go-generate card registration

Card definitions live in single-letter sub-packages under `mtg/cards/` (e.g., `mtg/cards/l/lightning_bolt.go`), grouped by the first letter of the card name. Each sub-package exports a `var Cards []*game.CardDef` slice generated by `go generate` (via `gencardlist`). The parent `mtg/cards.Registry` aggregates these slices. We chose alphabetical bucketing over flat-directory (too many files at scale), type-based grouping (merge conflicts, subjective boundaries), and `init()`-based registration (import cycles between sub-packages and parent). The `go generate` step means adding a card never requires editing any file beyond the card's own `.go` file — run `go generate ./mtg/cards/...` and the card list updates automatically.

## Considered Options

- **Flat directory** — all cards in `mtg/cards/*.go`. Simple but produces hundreds of files in one directory as the corpus grows.
- **Type-based grouping** — `mtg/cards/creatures.go`, `mtg/cards/instants.go`. Fewer files but subjective boundaries, large files, and merge conflicts.
- **`init()` registration** — each card file registers itself via `init()`. Clean but creates an import cycle between letter sub-packages and the parent registry.
- **Alphabetical sub-packages with go-generate (chosen)** — deterministic bucketing, small directories, no import cycles, no manual registry edits.
